# 15 AI 风险分析与评分引擎

## 目标

AI 风险引擎负责把 JD、HR 聊天、公司信息和用户反馈转化为可解释的风险报告。

它不是单纯调用大模型，而是由三部分组成：

```text
大模型结构化分析
  +
规则引擎校验
  +
评分公式计算
```

## 输入

```json
{
  "source_platform": "BOSS直聘",
  "company_name": "某某公司",
  "job_title": "储备主管",
  "jd_text": "岗位职责...",
  "hr_chat_text": "HR聊天内容，可为空",
  "company_snapshot": null,
  "feedback_summary": null
}
```

## 输出

```json
{
  "overall_score": 78,
  "risk_level": "高",
  "confidence": "中",
  "predicted_role": "销售/客户开发岗",
  "risk_types": ["管理岗包装销售岗", "薪资不透明"],
  "sub_scores": {
    "jd_risk": { "score": 82, "weight": 0.35 },
    "hr_risk": { "score": 75, "weight": 0.20 },
    "company_risk": { "score": null, "weight": 0.25, "status": "missing" },
    "feedback_risk": { "score": null, "weight": 0.20, "status": "missing" }
  },
  "strong_risk_adjustment": 8,
  "evidence": [
    "JD中出现“管理人才、实作期、市场实践、业绩分解”组合",
    "HR未正面回答是否有个人销售指标"
  ],
```

## Prompt 设计原则

必须要求模型：

- 只基于用户提供的信息判断
- 不做法律定性
- 不使用“骗子、黑公司、实锤”等词
- 每个风险结论必须有证据
- 缺少信息时输出 `missing_info`
- 输出固定 JSON
- 不输出 Markdown
- 不编造公司工商信息和用户反馈

## 评分逻辑

标准公式：

```text
综合风险分 =
JD 风险分 × 35%
+ HR 话术风险分 × 20%
+ 公司工商风险分 × 25%
+ 用户反馈风险分 × 20%
+ 强风险修正项
```

动态权重：

```text
最终风险分 =
Σ（已有子分 × 子模型权重）÷ Σ（已有子模型权重）
+ 强风险修正项
```

封顶 100。

## 风险等级

| 分数 | 等级 |
|---:|---|
| 0-30 | 低 |
| 31-60 | 中 |
| 61-80 | 高 |
| 81-100 | 极高 |

## 规则引擎兜底

### 强风险词

命中以下信号时，大模型分数不能低于中风险：

- 押金
- 保证金
- 培训贷
- 贷款分期
- 扣身份证
- 扣毕业证
- 先交费
- 拉亲友资源
- 无薪试岗

### 反误伤规则

以下情况不能直接判高风险：

- 岗位明确写销售，且薪资和指标说明清楚
- 初创公司成立时间短，但 JD 清晰、无收费、无回避
- HR 回复简短但没有回避核心问题
- 用户只提供很短 JD，证据不足

## 置信度规则

| 置信度 | 条件 |
|---|---|
| 高 | JD + HR + 工商/反馈 至少三类数据可用 |
| 中 | JD + HR 或 JD + 工商 可用 |
| 低 | 只有 JD，或 JD 太短 |

报告页必须展示置信度。

## HR 回复分析

HR 分析不重新生成完整报告，只回答：

- 是否正面回答用户问题
- 是否回避核心信息
- 是否提高/降低原报告风险
- 下一轮应该怎么问

输出：

```json
{
  "avoidance_score": 86,
  "risk_level": "高",
  "analysis": "HR未正面回答是否需要个人开发客户。",
  "next_questions": [
    "是否有个人销售指标？",
    "客户来源由公司提供，还是需要自己开发？"
  ]
}
```

## 测试集

上线前准备 50-100 条样本：

- 保险销售包装管理岗
- 正常销售岗
- 正常行政岗
- 培训贷
- 外包伪装正式岗
- 无薪试岗
- 初创公司正常岗位
- 高薪低门槛岗位
- HR 回避案例
- HR 正常解释案例

每条样本标注：

- 期望风险等级
- 期望命中风险类型
- 应生成的追问问题
- 不应出现的误判

## 技术可行性

AI 风险引擎可行，但必须避免完全依赖模型自由发挥。首版建议“大模型负责语义理解，规则引擎负责边界和修正”，这样能兼顾识别能力和可控性。
