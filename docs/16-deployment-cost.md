# 16 部署方案与成本评估

## 技术可行性结论

Web/H5 首版技术可行性高，建议立即进入工程实现规划。首版成本主要来自前后端开发和大模型 API 调用；OCR、工商查询和公司画像不应阻塞 MVP。

## 推荐部署方案

### 方案 A：低成本快速上线

```text
前端：Vercel / Cloudflare Pages
后端：轻量云服务器 / Serverless API
数据库：Supabase PostgreSQL / 云数据库
AI：第三方大模型 API
```

适合：

- MVP
- 内测
- 低流量灰度

优点：

- 上线快
- 运维少
- 成本低

缺点：

- Serverless 冷启动可能影响体验
- 后续 OCR 自建服务可能要单独部署

### 方案 B：单服务器 Docker Compose

```text
Nginx
  ↓
Frontend Static
Backend API
PostgreSQL
Redis
OCR Service，P1
```

适合：

- 需要更强控制
- 要自建 OCR
- 要统一部署

优点：

- 成本可控
- 架构清晰
- 自建 OCR 方便

缺点：

- 需要运维
- 扩容不如云服务方便

## 推荐首版

首版建议：

```text
前端：Vercel / Cloudflare Pages
后端：Node.js API 部署到轻量云服务器或 Render/Fly.io
数据库：Supabase PostgreSQL
AI：统一 LLM Provider
OCR：P1 后再单独部署
```

如果预计快速加入自建 OCR，可以直接选择单服务器 Docker Compose。

## 开发成本估算

| 模块 | 工期 |
|---|---:|
| Web/H5 前端 | 2-3 周 |
| 后端 API | 2-3 周 |
| AI 风险引擎 | 1-2 周 |
| 数据库与反馈 | 3-5 天 |
| 测试样本与调优 | 1 周 |
| OCR P1 | 1-2 周 |

如果 1 名前端 + 1 名后端 + 1 名产品/测试协作：

```text
MVP：3-5 周
P1 OCR：额外 1-2 周
```

## 主要技术风险

| 风险 | 影响 | 应对 |
|---|---|---|
| AI 输出不稳定 | 报告渲染失败或误判 | JSON Schema 校验 + 重试 + 规则引擎 |
| AI 成本失控 | 调用量上升后费用变高 | 限流、缓存、文本长度限制 |
| 高风险误判 | 用户或企业投诉 | 证据展示、误判反馈、免责声明 |
| OCR 识别不准 | 字段抽取错误 | 用户确认页、文本补充入口 |
| 隐私泄露 | 法律和信任风险 | 不公开原文、短期保存、删除机制 |
| 移动端输入体验差 | 完成率低 | 示例提示、分段输入、P1 截图 OCR |

## 技术 Go / No-Go

### 建议 Go

满足以下条件即可进入开发：

- 首版只做文本检测主路径
- 风险报告 JSON 协议固定
- 有 50 条以上测试样本
- 有隐私说明和免责声明
- 有误判反馈入口

### 暂不建议 Go

如果坚持首版同时做以下内容，则不建议立即开发：

- 工商 API
- 自建 OCR
- 公司画像
- 社区
- 企业申诉后台
- 用户登录与会员

这些会显著拉长周期，降低 MVP 验证速度。

## 最终建议

技术专家建议：

```text
先做 Web/H5 文本检测 MVP。
并行预研 OCR，但不阻塞首版上线。
```

第一版目标不是做一个完整招聘风控系统，而是验证用户是否愿意把真实岗位信息交给产品，并认为风险报告和追问卡片有用。
