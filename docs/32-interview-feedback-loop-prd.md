# JobLens 面试反馈闭环 P0 产品需求

版本：v1.0

日期：2026-07-15

状态：产品评审通过，待开发

## 1. 背景

当前面试反馈可以记录公司、岗位、JD 描述、实际面试内容和少量风险标记，但尚未形成可用于风险分析的数据闭环：

- 反馈可以不关联原报告，岗位匹配准确度不足。
- 风险标记混合事实和主观感受，脏数据可能放大误判。
- 缺少重复内容、批量举报和关联冲突治理。
- 没有单条反馈权重、独立有效样本量和聚合置信度。
- 风险引擎无法区分模型推断与真实面试反馈。

本次迭代将面试反馈从“匿名留言”升级为“结构化、可审核、可聚合的风险信号”。

## 2. 产品目标

1. 用户能在一分钟内完成一次与原报告关联的面试反馈。
2. 无效、重复、恶意和纯情绪反馈不得影响风险分析。
3. 单条反馈不能直接改变岗位结论，必须经过独立样本聚合。
4. 风险报告清楚区分“模型推断”和“用户反馈验证”。
5. 每次聚合结果可解释、可追溯、可撤回，不直接在线训练模型。

核心链路：

```text
岗位分析 -> 面试后回访 -> 结构化反馈 -> 硬过滤
-> 单条权重 -> 同岗位聚合 -> 风险分析辅助信号 -> 用户再次反馈
```

## 3. 非目标

- P0 不建设公开评论区、公司黑名单或用户互动社区。
- P0 不自动公开反馈原文、证据图片或反馈者身份。
- P0 不将用户反馈用于实时微调或自动修改模型 Prompt。
- P0 不依据单条反馈给企业或岗位作“诈骗”“骗子”等法律定性。
- P0 不引入付费内容审核服务和复杂账号体系。

## 4. 用户流程

### 4.1 入口

优先入口位于报告页“面试后验证”模块，自动携带 `report_id`。首页保留独立入口，但未关联报告的反馈只能进入待匹配池，P0 不参与风险聚合。

报告页入口状态：

- 未反馈：`面试后回来验证`
- 已反馈：`已提交，审核中`
- 已纳入：`已纳入匿名统计`
- 被过滤：只提示用户“该反馈暂未纳入统计”，不公开反作弊细节

### 4.2 表单步骤

第一步，确认经历：

- 已参加面试
- 沟通后未面试
- 发现异常后放弃
- 已入职或试岗

第二步，确认一致性：

- 基本一致
- 部分不一致
- 完全不是同一岗位
- 无法判断

第三步，选择事实型风险标签：

- 实际工作涉及销售，但 JD 未说明
- 到场后更换岗位
- 固定底薪或薪资结构与 JD 不符
- 要求缴纳费用、押金或购买产品
- 诱导贷款、分期或付费培训
- 要求发展下线、拉人或使用亲友资源
- 招聘、面试、合同或社保主体不一致
- 扣留或过度索取身份证、银行卡等信息
- 办公地址或公司主体异常
- 其他可描述事实

第四步，填写事实描述和证据等级：

- 仅个人判断
- 有具体过程描述
- 招聘方明确口头说明
- 有聊天记录或现场材料
- 有合同、收费凭证等材料

第五步，可选体验评价：推荐、中立、不推荐。该字段只用于体验统计，不进入岗位风险分。

### 4.3 提交结果

提交成功文案：

> 反馈已匿名提交。系统会进行重复性、关联性和内容质量检查，通过后才会纳入岗位风险统计。

不得承诺“提交后立即优化模型”或“举报立即生效”。

## 5. 数据分类

| 数据类型 | 示例 | 是否参与风险计算 |
|---|---|---|
| 可核验事实 | 收取 500 元培训费、实际要求销售保险 | 是 |
| 事实上下文 | 面试时间、经历阶段、平台、岗位 | 用于关联和权重 |
| 证据等级 | 具体描述、聊天材料、合同凭证 | 用于权重 |
| 主观体验 | HR 态度差、不推荐 | 否 |
| 敏感信息 | 姓名、手机、身份证、银行卡 | 脱敏后再处理 |
| 情绪和定性 | 骗子、黑公司、垃圾 | 不参与计算 |

### 5.1 反馈来源与自选择偏差

主动进入反馈页的用户更可能经历了负面事件，不能把主动反馈中的负面占比解释为“所有候选人的遇到概率”。反馈必须区分来源：

| 来源 | 定义 | 用途 |
|---|---|---|
| neutral_prompt | 系统对符合条件的报告用户进行中性邀请后提交 | 可以同时贡献支持和否认样本 |
| report_voluntary | 用户从报告页主动提交 | 可证明风险事件存在，但不得单独用于估算发生率 |
| homepage_voluntary | 用户从首页独立入口提交 | P0 不进入聚合 |

系统不得通过提高“好评”权重来机械平衡差评。解决偏差依赖中性采样、降低正向反馈成本和降低低覆盖样本的置信度。

中性邀请要求：

- 对满足条件的报告按固定比例随机展示，不根据原报告风险高低决定是否邀请。
- 首问使用中性文案：“后续沟通或面试中，实际情况与岗位描述一致吗？”
- “基本一致”和“没有遇到以上情况”必须是一键可提交的低成本路径。
- 正向和负向反馈获得相同权益，不以选择风险标签作为奖励条件。
- 记录邀请曝光和响应，未响应不计为正向或负向样本。
- 同一报告最多邀请一次，允许用户选择“尚未面试”或“不再提醒”。

## 6. 质量治理

### 6.1 硬过滤

满足任一条件时，反馈权重为 0，不进入聚合：

- 未关联有效报告或报告已删除。
- 同一 visitor 对同一报告和事实标签重复提交。
- 内容或证据哈希与已有反馈高度重复，且来源不独立。
- 只有情绪定性，没有可识别的经历或事实。
- 面试时间、招聘时间或经历阶段存在不可解释的逻辑冲突。
- Turnstile、短窗口限流或敏感数据检查不通过。
- 同设备、IP 或文本簇在短时间内集中举报同一对象。
- 内容与岗位、公司或招聘过程无关。
- 人工审核确认为恶意、伪造或广告内容。

包含敏感信息不直接作废；能够可靠脱敏时进入待审核，否则拒绝入库并提示用户修改。

### 6.2 单条反馈权重

所有分量范围为 0 到 1：

```text
W = clamp(
  G * M * T * I * A * S
  * (0.15 + 0.20Q + 0.30E + 0.20R + 0.15C),
  0,
  1
)
```

| 符号 | 含义 | 规则 |
|---|---|---|
| G | 硬过滤门 | 有效为 1，无效为 0 |
| M | 岗位关联度 | 原报告精确关联 1.0；人工匹配最高 0.7；未匹配 0 |
| T | 时间有效性 | `max(0.3, exp(-days / 180))` |
| I | 独立性 | 独立样本 1.0；同网络 0.7；相似内容 0.3；同设备重复 0.1 |
| A | 反作弊系数 | `1 - abuse_score` |
| S | 采样来源系数 | 中性邀请 1.0；报告页主动提交 0.7；首页未匹配 0 |
| Q | 信息完整度 | 有效关键字段数 / 关键字段总数 |
| E | 证据强度 | 个人判断 0.2；过程描述 0.5；明确说明 0.6；已核验材料 0.8；已核验凭证 1.0 |
| R | 反馈者可信度 | `(历史有效数 + 3) / (历史已判定总数 + 6)` |
| C | 内容一致性 | 结构化标签、否认项、事实描述和原 JD 的规则一致程度 |

设计约束：

- 新 visitor 的 `R` 为 0.5，既不被完全忽略，也不获得最高权重。
- `M=0` 或 `G=0` 时最终权重必须为 0。
- P0 尚未上传并核验材料，`E` 最高为 0.6；用户自述“有截图”不等于已核验证据。
- 后续证据文件本身也不自动获得高权重，仍需相关性、脱敏和重复性检查。
- `R` 只计算已接受或已拒绝的历史反馈，`pending` 不进入分母。
- `C` 使用可审计规则计算；例如同时支持和否认同一标签时降为 0，禁止用模型主观判真伪。
- `S` 只修正样本用途和影响上限，不能消除主动反馈中未知的选择偏差。
- 计算过程保留分量和规则版本，支持复算与申诉撤回。

## 7. 聚合规则

每个风险标签独立聚合，禁止把“不推荐”当作全部风险标签的支持票。

```text
P = 支持该风险标签的反馈权重之和
N = 明确否认该风险标签的反馈权重之和
RiskConfidence = (1 + P) / (2 + P + N)
EffectiveN = (sum(W) ^ 2) / sum(W ^ 2)
Disagreement = 1 - abs(P - N) / (P + N)
PromptedShare = neutral_prompt 有效权重 / 全部有效权重
ResponseQuality = min(1, 同邀请批次近 30 天响应率 / 0.20)
SamplingQuality = min(1, PromptedEffectiveN / 3) * sqrt(ResponseQuality)
AdjustedConfidence = 0.5 + (RiskConfidence - 0.5) * SamplingQuality
```

当 `P + N = 0` 时，风险标签状态为 `insufficient`，争议度不计算。

`RiskConfidence` 只表示反馈者样本内部的一致程度，不等于现实发生率。对外和风险引擎统一使用向 0.5 收缩后的 `AdjustedConfidence`。未响应邀请不得作为否认风险的样本；低响应率只能降低采样质量，不能被解释为正向反馈。20% 是 P0 的目标响应率，应通过配置和版本管理调整，而不是散落在业务代码中。

| 状态 | 最低条件 | 产品处理 |
|---|---|---|
| insufficient | 独立反馈少于 2 | 仅内部记录 |
| emerging | 独立反馈不少于 2，P 不少于 1.0 | 报告可提示“收到相关反馈”，不展示发生率、不加分 |
| supported | 独立反馈不少于 3，EffectiveN 不少于 2.5，至少 1 个中性邀请样本，AdjustedConfidence 不低于 0.65 | 注入风险分析，反馈子分最高 60 |
| strong | 独立反馈不少于 5，EffectiveN 不少于 4，PromptedEffectiveN 不少于 2，AdjustedConfidence 不低于 0.75 | 反馈子分可高于 60，报告重点展示 |
| disputed | 争议度不低于 0.6 | 只显示“反馈存在分歧”，不得作高风险修正 |

收费、贷款、扣证件等强风险仍需至少 2 个独立样本，其中至少 1 条已核验反馈 `E >= 0.8`，才允许进入重点提示。由于 P0 不提供证据上传与核验，这些标签在 P0 只能形成核实建议，不得仅凭反馈触发强风险结论。单条材料在后续版本中也只能触发人工审核，不能直接触发强风险结论。

## 8. 反哺风险分析

风险引擎只接收聚合快照，不接收原始反馈正文：

```json
{
  "risk_tag": "DISGUISED_SALES",
  "independent_reporters": 6,
  "effective_sample_size": 4.8,
  "positive_weight": 4.3,
  "negative_weight": 0.7,
  "confidence": 0.78,
  "disagreement": 0.28,
  "evidence_level": "medium",
  "status": "strong",
  "weight_rule_version": "feedback-weight-v1"
}
```

回灌原则：

- `insufficient` 不进入模型输入。
- `emerging` 只生成追问建议，不改变综合风险分。
- `supported` 和 `strong` 进入 `feedback_risk` 子分，最高占综合风险分 20%。
- 用户反馈不能单独把综合结论推到“极高风险”。
- 高争议数据不加风险分，只提醒用户主动核实。
- 报告必须展示样本数、证据等级和“用户反馈”来源。
- 主动反馈占主导时，只展示“收到若干独立反馈”，禁止展示“负面率”“多数候选人”等比例性结论。

## 9. 合规与申诉

- 对外展示聚合统计，不展示反馈者身份、IP、原文和未脱敏证据。
- 使用“面试反馈显示”“存在岗位信息不一致风险”，禁止使用“实锤”“骗子公司”等表述。
- 企业主体、反馈者或管理员撤回有效反馈后，聚合结果必须在 24 小时内复算。
- 原始反馈按隐私策略保留，聚合快照不得绕过删除请求保留可识别内容。
- 管理后台必须能查看规则命中原因，但普通用户看不到反作弊阈值。

## 10. 指标

核心指标：

- 报告关联反馈率。
- 中性邀请曝光量、响应率、正向/负向完成率和两者表单退出率差异。
- 中性邀请样本占比与 `PromptedEffectiveN`。
- 表单提交完成率和中位填写时长。
- 硬过滤率、重复率、待审核率、有效反馈率。
- 独立有效样本量达到 3 的岗位数。
- 聚合信号与后续新增反馈的一致率。
- 因反馈信号发生风险等级变化的报告比例。
- 被撤回、申诉成立和误伤率。

护栏指标：

- 单一 visitor、IP、设备簇贡献的有效权重占比。
- 高风险结论中缺少独立样本或证据的比例，目标为 0。
- 原始反馈或敏感信息对外泄露事件，目标为 0。

## 11. P0 范围与验收

P0 必做：

1. 报告页携带 `report_id` 进入反馈页，并自动填充公司、岗位和 JD。
2. 新增经历阶段、一致性、事实风险标签、证据等级和面试日期。
3. 对固定比例的报告展示中性邀请，并记录曝光、响应和来源。
4. “基本一致/未遇到异常”提供不超过两步的一键反馈路径。
5. 未关联报告的反馈不进入自动聚合。
6. 实现重复、内容质量、逻辑冲突和集中提交硬过滤。
7. 持久化采样来源、权重分量、最终权重、规则版本和审核状态。
8. 按报告输入哈希和风险标签生成聚合快照。
9. `supported` 以上快照进入风险分析，报告展示反馈来源和样本信息。
10. 反馈撤回或审核状态变化后可重新计算。
11. 完成中英文界面和接口枚举兼容。
12. 权重、采样质量和聚合算法具备固定样例单元测试。

P1 延后：证据图片上传和自动脱敏、公司主体归一化、跨平台同岗聚类、运营审核后台、企业申诉入口。

## 12. 产品评审结论

结论：有条件通过，可以进入技术评审。

冻结决策：

- P0 只允许与原报告强关联的反馈影响风险分析。
- 主观推荐不参与风险权重。
- 使用规则计算权重，不使用大模型直接给反馈定真假。
- 单条反馈永不直接改变风险等级。
- 主动反馈只能作为事件信号，中性邀请样本达到门槛后才能形成比例性判断。
- 先做岗位级反馈闭环，暂不做公司级黑名单和公开评论。

进入开发前必须由技术评审确认：身份独立性边界、数据库迁移方案、聚合一致性、历史反馈处理和测试基准。
