# 23 Test Samples Plan - P0 回归集

冻结日期：2026-07-13
版本：v1.0.0

## 说明

本文档冻结 JobLens Web/H5 MVP P0 阶段测试样本集规划。上线前必须准备至少 50 条脱敏真实岗位样本，用于：

1. AI 模型输出质量回归测试
2. 规则引擎修正逻辑验证
3. 前端报告页渲染兼容性测试
4. 风险评分准确性人工评估
5. 误伤和漏检样本识别

测试样本来源：
- 公开招聘平台真实岗位信息（已脱敏）
- 用户面试反馈样本（已脱敏）
- 典型风险案例构造样本
- 正常岗位对比样本

**所有样本必须脱敏处理，不得包含真实姓名、完整手机号、身份证号、银行卡号等敏感信息。**

## 样本分类与数量目标

### 1. 总体目标

| 类别 | 数量目标 | 说明 |
|---|---:|---|
| 高风险样本 | 15-20 条 | overall_score ≥60，必须包含强风险证据 |
| 中风险样本 | 15-20 条 | overall_score 31-60，部分风险信号 |
| 低风险样本 | 15-20 条 | overall_score ≤30，正常岗位或信息不足 |
| 极高风险样本 | 5-10 条 | overall_score ≥80，涉及违法或收费 |
| **总样本数** | **50-70 条** | 足够覆盖主要场景 |

---

### 2. 高风险样本类别

| 风险类型 | 数量目标 | 典型特征 | 验证标准 |
|---|---:|---|---|
| 管理岗包装销售岗 | 5-8 条 | JD 提管理，实际销售保险/金融产品 | predicted_role 为"销售岗"，overall_score ≥60 |

## 样本标注要求

每个样本必须标注：

| 标注字段 | 说明 | 示例 |
|---|---|---|
| sample_id | 样本唯一标识 | `sample_001` |
| risk_type | 风险类型分类 | `管理岗包装销售岗` |
| expected_score | 期望风险分范围 | `61-80` |
| expected_level | 期望风险等级 | `高` |
| expected_risk_types | 期望命中的风险类型列表 | `["管理岗包装销售岗", "薪资不透明"]` |
| expected_evidence | 期望证据关键词 | `"管理人才"、"实作期"` |
| expected_questions | 期望追问关键词 | `"是否有个人销售指标"` |
| forbidden_judgments | 不应出现的误判 | `"正常销售岗"`（对于包装岗样本） |
| input_jd | JD 文本（已脱敏） | 见下文示例 |
| input_hr_chat | HR 聊天文本（可选，已脱敏） | 见下文示例 |

## 回归测试流程

### 1. 自动化回归测试

每次 AI 模型或规则引擎调整后，必须运行自动化回归测试：

```text
读取所有样本
  ↓
调用 POST /api/reports/detect（使用 MockProvider 或真实 Provider）
  ↓
校验输出是否符合 expected_score、expected_level、expected_risk_types
  ↓
检查是否出现 forbidden_judgments
  ↓
记录所有不符合样本
  ↓
生成回归测试报告
  ↓
不符合样本数 > 5% → 阻止上线，调整模型或规则
```

### 2. 人工质量评估

上线前必须进行人工质量评估：

| 评估维度 | 方法 | 门槛 |
|---|---|---|
| 高风险准确性 | 人工复核高风险样本报告 | 高风险样本不得误判为低风险 |
| 低风险准确性 | 人工复核低风险样本报告 | 正常销售岗不得误判为包装岗 |
| 证据充分性 | 人工检查高风险报告证据 | 高风险报告必须有明确证据 |
| 追问有效性 | 人工检查追问问题是否可执行 | 追问问题必须明确、可执行 |
| 表达合规性 | 人工检查敏感表达过滤 |不得出现"骗子、黑公司、实锤"等表达 |

## 样本脱敏规则

所有样本必须脱敏处理：

| 脱敏内容 | 处理方式 |
|---|---|
| 真实姓名 | 替换为"张先生"、"李女士"或虚构姓名 |
| 完整手机号 | 替换为 `138****5678` 或虚构号码 |
| 身份证号 | 完全删除或替换为虚构号码 |
| 银行卡号 | 完全删除或替换为虚构号码 |
| 真实公司名称 | 可保留真实公司名（公司信息不视为个人隐私），或虚构公司名 |
| 真实岗位名称 | 可保留真实岗位名 |

## 样本存储与管理

### 1. 样本文件存储

样本存储在 Git 仓库中：

```text
docs/
  test-samples/
    sample_001.json
    sample_002.json
    ...
    sample_050.json
```

每个样本为一个 JSON 文件，格式见上文示例。

### 2. 样本版本管理

样本集必须版本管理：

| 版本 | 说明 |
|---|---|
| v1.0.0 | P0 MVP 冻结版本，50-70 条样本 |
| v1.1.0 | 新增样本或调整标注，记录变更原因 |
| v2.0.0 | P1 OCR 样本或新增风险类型 |

每次调整样本集必须：
1. 更新样本文件
2. 运行回归测试验证
3. 更新版本号和变更记录

## 后端实现要点

1. 实现自动化回归测试脚本

4. 前端必须处理极端值（如 `overall_score` 为 0 或 100）
