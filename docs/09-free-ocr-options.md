# 09 免费 OCR 与项目内 OCR 模块方案

## 结论

如果目标是降低长期调用成本，建议优先考虑“项目后端自建 OCR 服务”，而不是在微信小程序端直接跑 OCR。

推荐路径：

```text
MVP：文本粘贴为主，截图 OCR 不阻塞上线
  ↓
P1：后端自建 OCR 服务，优先评估 RapidOCR / PaddleOCR
  ↓
P2：自建 OCR + 云厂商 OCR 兜底
```

不建议首版在小程序端直接跑 OCR。原因是小程序包体、运行性能、中文模型体积、WASM 兼容和识别速度都不稳定。

## 方案对比

| 方案 | 是否免费 | 适合端 | 难度 | 成本 | 建议 |
|---|---|---|---|---|---|
| PaddleOCR 自建服务 | 开源免费 | 后端服务 | 中高 | 服务器成本 | 推荐 P1 评估 |
| RapidOCR 自建服务 | 开源免费 | 后端服务 | 中 | 服务器成本 | 推荐优先试 |
| Tesseract OCR | 开源免费 | 后端 / H5 | 中 | 服务器或前端性能成本 | 可作为备选 |
| Tesseract.js | 开源免费 | Web/H5 | 中 | 用户设备性能成本 | H5 可试，小程序不推荐 |
| 云厂商 OCR | 按量收费 | 后端调用 | 低 | API 调用费 | 适合作兜底 |
| 小程序端本地 OCR | 理论可行 | 小程序 | 高 | 开发和性能成本 | 不建议首版 |

## 推荐方案一：后端自建 RapidOCR

RapidOCR 是轻量 OCR 方案，适合先做工程验证。它的优势是部署成本相对低，速度和资源占用通常比完整 PaddleOCR 更友好。

### 适合场景

- 岗位截图
- HR 聊天截图
- 普通中文截图
- 移动端网页截图

### 推荐架构

```text
小程序上传截图
  ↓
云存储 / 后端临时文件
  ↓
OCR 服务识别
  ↓
返回 raw_ocr_text
  ↓
大模型结构化岗位字段
  ↓
用户确认
  ↓
风险检测
```

### 优点

- 不按次付 OCR API 费用
- 可控性强
- 可以本地调试
- 后期可以针对招聘截图优化预处理

### 缺点

- 需要服务器或容器
- 需要维护模型和运行环境
- 首次部署比云 API 麻烦
- 识别效果需要用真实岗位截图测试

## 推荐方案二：后端自建 PaddleOCR

PaddleOCR 更成熟，中文场景效果通常较好，但部署比轻量方案重。

### 优点

- 中文识别能力强
- 社区成熟
- 可扩展表格、版面分析等能力

### 缺点

- 依赖较多
- 模型较大
- 云函数环境不一定适合
- 更适合部署在轻量云服务器、容器服务或 GPU/CPU 服务器

## 方案三：H5 使用 Tesseract.js

如果未来做 Web/H5，可以考虑在浏览器端用 Tesseract.js 做免费 OCR。但这不适合小程序首版。

### 优点

- 不需要后端 OCR 调用费
- 图片不必上传服务器，隐私更好
- H5 可快速试验

### 缺点

- 中文识别模型较大
- 首次加载慢
- 手机上识别速度不稳定
- 识别准确率不一定适合招聘截图
- 小程序环境兼容和包体限制不友好

## 为什么不建议小程序端直接跑 OCR

小程序端直接跑 OCR 会遇到几个问题：

- 中文 OCR 模型体积大，影响小程序包体和加载速度
- 手机性能差异大，低端机识别慢
- 长截图和多图识别容易卡顿
- WASM/Worker 能力受限，调试成本高
- 识别失败时用户体验差

因此，小程序更适合做图片选择、压缩、上传和结果确认，OCR 运算放在后端。

## 成本判断

### 云厂商 OCR 成本

按量调用简单，但量起来后持续付费。适合早期快速验证和后端自建失败时兜底。

### 自建 OCR 成本

自建 OCR 没有按次调用费，但需要服务器成本。

粗略估算：

| 阶段 | 方案 | 月成本 |
|---|---|---:|
| 内测 | 本地或低配服务器跑 OCR | 0-200 元 |
| 小规模灰度 | 轻量服务器 / 容器 | 100-500 元 |
| 稳定增长 | 独立 OCR 服务 + 队列 | 500-2000 元 |
| 高并发 | 多实例 + 缓存 + 云 OCR 兜底 | 视调用量而定 |

## 推荐落地策略

### 第一步：先做 OCR 服务接口，不绑定具体供应商

接口保持统一：

```json
{
  "image_file_ids": ["cloud://xxx/ocr_001.jpg"],
  "ocr_provider": "self_hosted",
  "raw_ocr_text": "...",
  "confidence": "中",
  "debug": {
    "engine": "rapidocr",
    "duration_ms": 1200
  }
}
```

这样后面可以在 `self_hosted`、`aliyun`、`tencent` 之间切换。

### 第二步：先接自建 RapidOCR 做验证

验证 50-100 张真实岗位截图：

- BOSS 岗位详情页
- 智联岗位详情页
- 猎聘岗位详情页
- 微信聊天截图
- 小红书招聘截图
- 模糊截图
- 长截图裁剪图

### 第三步：失败时走云 OCR 兜底

当自建 OCR 识别为空、耗时过长或置信度低时，再调用云 OCR。

推荐策略：

```text
自建 OCR 成功 → 直接进入结构化
自建 OCR 失败 → 调用云 OCR
云 OCR 仍失败 → 提示用户粘贴文本
```

## 对产品文案的影响

上传入口建议写：

```text
上传岗位截图
识别后可修改，建议上传清晰的岗位详情页或 HR 回复截图。
```

识别失败时写：

```text
截图文字识别不完整，建议补充粘贴岗位描述。
```

不要承诺“自动准确识别所有截图”。

## 工程建议

新增模块：

```text
cloudfunctions/
  ocr-extract/
    index.js 或 index.py

services/
  ocr/
    provider.ts
    self-hosted.ts
    aliyun.ts
    fallback.ts
```

如果使用微信云开发，云函数本身不一定适合跑重 OCR 模型。更稳妥的是：

```text
小程序云函数负责鉴权和调度
独立 OCR 服务负责图片识别
```

## 最终建议

优先级建议调整为：

| 阶段 | OCR 策略 |
|---|---|
| MVP | 不强依赖 OCR，文本粘贴为主 |
| P1 | 自建 RapidOCR / PaddleOCR 服务 |
| P1 兜底 | 云厂商 OCR 低置信度兜底 |
| P2 | 针对招聘平台截图做版面优化和字段抽取 |

如果你希望“尽量免费”，最合理的是后端自建 OCR，而不是小程序端本地 OCR。这样既能控制长期成本，也能保证小程序端体验稳定。
