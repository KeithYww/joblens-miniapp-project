# 08 上传岗位截图提取岗位信息调研

## 需求结论

“上传岗位截图并提取岗位信息”建议加入产品规划，但不建议作为 MVP 的唯一输入方式。

首版仍以“粘贴 JD 文本”为主，截图识别作为 P1 增强能力进入第一阶段后半段或第二阶段。原因是截图识别能显著降低用户输入成本，但会引入图片上传、OCR 调用、结构化抽取、隐私处理和识别纠错等额外复杂度。

## 用户价值

岗位截图识别适合以下场景：

- 用户在 BOSS、智联、猎聘、小红书、微信群里看到岗位，不方便复制完整 JD
- HR 聊天记录较长，用户更愿意截图上传
- 用户想一次性上传岗位页、薪资页、HR 回复页
- 移动端用户不愿意在小程序里长按复制多段文本

对用户来说，截图识别的价值不是“更高级”，而是更省事：

```text
上传截图 → 自动识别岗位名称、公司、薪资、职责、HR 话术 → 用户确认 → 生成风险报告
```

## 技术可行性

微信小程序侧具备基础图片选择和上传能力。`wx.chooseMedia` 支持拍摄或从手机相册选择图片，返回本地临时文件路径和文件大小；`wx.uploadFile` 可将本地资源通过 HTTPS POST 上传到服务器；如果使用微信云开发，也可以用 `wx.cloud.uploadFile` 上传到云存储并返回 `fileID`。  

参考资料：

- 微信 `wx.chooseMedia` 官方文档：https://developers.weixin.qq.com/miniprogram/dev/api/media/video/wx.chooseMedia.html
- 微信 `wx.uploadFile` 官方文档：https://developers.weixin.qq.com/miniprogram/dev/api/network/upload/wx.uploadFile.html
- 微信云开发 `wx.cloud.uploadFile` 官方文档：https://developers.weixin.qq.com/miniprogram/dev/wxcloud/reference-sdk-api/storage/uploadFile/client.uploadFile.html

OCR 能力可接云厂商通用文字识别。阿里云通用文字识别官方说明支持识别图片、文档、扫描件、手写件中的文字信息，并可返回通用文字识别结果；其页面显示通用文字识别有免费额度，并给出按量付费价格区间。  

参考资料：

- 阿里云通用文字识别：https://ai.aliyun.com/ocr/general
- 腾讯云 OCR 文字识别介绍：https://cloud.tencent.com/act/pro/ocr-for-old-users

## 推荐技术流程

```text
用户上传截图
  ↓
小程序本地校验图片大小与格式
  ↓
上传到云存储或业务后端
  ↓
调用 OCR 提取图片文字
  ↓
调用大模型进行岗位信息结构化
  ↓
用户确认或修改识别结果
  ↓
进入现有 detectJobRisk 风险检测流程
```

## 结构化字段

OCR 只负责“图片转文字”，不能直接保证岗位字段准确。OCR 之后还需要结构化抽取。

建议抽取字段：

| 字段 | 说明 |
|---|---|
| company_name | 公司名称 |
| job_title | 岗位名称 |
| salary_range | 薪资范围 |
| location | 工作地点 |
| experience_requirement | 经验要求 |
| education_requirement | 学历要求 |
| job_responsibilities | 岗位职责 |
| job_requirements | 任职要求 |
| benefits | 福利 |
| hr_chat_text | HR 聊天内容 |
| raw_ocr_text | OCR 原文 |

## 实现难度

| 模块 | 难度 | 说明 |
|---|---|---|
| 图片选择与上传 | 低 | 小程序已有标准 API，主要处理权限、大小、进度和失败重试 |
| 图片压缩 | 中 | 需要平衡上传速度和文字清晰度，不能压得太糊 |
| OCR 调用 | 中 | 云厂商 API 接入不难，但要处理费用、失败、超时、限流 |
| 岗位字段结构化 | 中高 | OCR 文本可能乱序，需要大模型抽取字段并让用户确认 |
| 多截图合并 | 中高 | 用户可能上传岗位页、详情页、聊天页多张图，需要合并上下文 |
| 隐私与安全 | 中高 | 截图可能包含手机号、微信号、头像、聊天昵称，需要脱敏与删除策略 |
| 识别纠错体验 | 中 | 必须允许用户修改识别出来的岗位信息 |

综合判断：如果只做“单图 OCR + 结构化确认”，实现难度为中等；如果做“多图合并 + 自动字段归类 + 隐私脱敏 + 历史识别”，难度会上升到中高。

## 成本估算

### 开发成本

| 版本 | 功能 | 工期 | 外包成本粗估 |
|---|---|---:|---:|
| 极简版 | 单图上传 + OCR + 文本回填 | 3-5 天 | 0.5-1.5 万 |
| 标准版 | 多图上传 + OCR + 结构化字段确认 | 1-2 周 | 2-5 万 |
| 完整版 | 多图合并、隐私脱敏、识别纠错、历史记录 | 2-4 周 | 5-10 万 |

### 调用成本

阿里云通用文字识别页面显示，通用文字识别有 200 次/月免费额度；按量计费中，通用文字识别在 1 万次以下档位显示为 0.0825 元/次，1 万到 10 万次档位显示为 0.0495 元/次。全文识别高精版在 1 万次以下档位显示为 0.225 元/次。具体以云厂商实际订单页为准。

按阿里云通用文字识别价格估算：

| 月识别量 | 按 0.0825 元/次估算 | 说明 |
|---:|---:|---|
| 1,000 次 | 82.5 元 | 早期测试成本可接受 |
| 10,000 次 | 825 元 | 小规模灰度可接受 |
| 50,000 次 | 约 2,475-4,125 元 | 根据阶梯价和接口类型变化 |
| 100,000 次 | 约 4,950-8,250 元 | 需考虑资源包或商务折扣 |

注意：实际成本还包括云存储、图片上传流量、大模型结构化调用和失败重试。

## 方案比较

| 方案 | 优点 | 缺点 | 建议 |
|---|---|---|---|
| 云厂商 OCR + 大模型结构化 | 稳定、开发快、准确率较好 | 有调用成本，需处理隐私 | 推荐首选 |
| 只用多模态大模型识图 | 可直接理解截图上下文 | 成本可能更高，稳定性和审计性较弱 | 可作为后续增强 |
| 自建 OCR | 长期成本可控 | 初期开发和维护成本高 | 不适合 MVP |
| 只让用户粘贴文本 | 成本最低 | 移动端输入麻烦 | MVP 必保留 |

## 推荐首版实现

第一版建议采用“文本粘贴为主，截图识别为辅”的设计：

1. 检测页默认展示文本输入框。
2. 增加“上传岗位截图”入口，但标注“识别后可修改”。
3. 首版只支持 1-3 张图。
4. OCR 后进入“识别结果确认页”。
5. 用户确认后再进入风险检测。
6. 图片识别完成后默认删除原图，只保留 OCR 文本和结构化字段。

## 产品交互建议

### 上传页

文案：

```text
上传岗位截图，系统会先识别文字，再生成风险报告。
请避免上传身份证、银行卡等敏感信息。
```

限制：

- 最多 3 张
- 单张建议不超过 5MB
- 支持岗位详情页、薪资页、HR 聊天截图

### 识别确认页

必须让用户确认：

- 公司名称是否正确
- 岗位名称是否正确
- 薪资是否正确
- 职责是否完整
- 是否包含 HR 聊天内容

如果识别结果不完整，提示：

```text
截图内容可能不完整，建议补充粘贴岗位描述或继续上传详情页。
```

## 隐私与合规要求

截图可能包含头像、昵称、手机号、微信号、定位、聊天记录等敏感信息。必须做到：

- 上传前提示用户不要上传身份证、银行卡等敏感图片
- OCR 后默认删除原图或设置短期自动过期
- 报告不公开展示截图原图
- 反馈公开时不展示完整聊天原文
- 后台访问图片需要权限控制
- 支持用户请求删除图片与识别结果

## 最终建议

该需求值得做，但不应阻塞 MVP。

建议排期：

| 阶段 | 建议 |
|---|---|
| MVP 第 1 版 | 保留粘贴文本输入，不做截图 OCR |
| MVP 第 2 版 | 增加单图/三图 OCR，识别后用户确认 |
| Beta 阶段 | 增加多图合并、隐私脱敏、识别纠错 |
| 增长阶段 | 结合截图识别做“岗位页一键检测” |

如果要尽快体验，可以先做标准版：多图上传、OCR、结构化字段确认。预计 1-2 周可以完成，开发成本约 2-5 万，调用成本早期每月通常在数百元级别，具体取决于识别量、选择的 OCR 服务和大模型调用次数。
