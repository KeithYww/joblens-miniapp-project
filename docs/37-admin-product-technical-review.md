# JobLens 管理后台多方评审纪要

日期：2026-07-15

## 1. 产品方案评审

| 角色 | 关注点 | 结论 |
|---|---|---|
| 产品负责人 | 数据必须能指导模型、成本和反馈决策 | 采用四工作台，指标必须可下钻到报告或反馈 |
| 运营负责人 | 用户反馈需要真正形成待办闭环 | 一期必须支持状态筛选、审核和内部备注 |
| 数据负责人 | 指标口径要稳定，避免模型调用与缓存混淆 | 报告维度看模型来源，API 维度看实际 `ai_called` |
| 安全负责人 | 后台会放大数据访问风险 | 独立 Admin Token、同源 CORS、无 IP/visitor 输出、审计审核动作 |
| 用户隐私负责人 | 原始 JD 可能包含招聘方信息 | 仅管理员可见，遵循现有 7 天源文本清理；不延长数据保留期 |

产品评审一致通过 P0 范围。复杂 RBAC、导出、批量操作、模型配置、公司风险库和反馈自动反哺模型进入后续版本。

## 2. 技术评审

### 2.1 方案选择

- 管理页面集成到现有 React/Vite 应用的 `/admin`，复用 Vercel 与现有设计系统。
- 后端在现有 Fastify 服务增加 `/api/admin/*`，复用 Prisma、Redis 和安全响应头。
- 使用独立 `ADMIN_TOKEN` Bearer 鉴权，不与监控、备份 Token 复用。
- 聚合查询直接基于当前数据量执行；列表统一分页且限制最大页大小。
- 不新增数据库迁移，复用反馈审核字段和 SecurityEvent。

### 2.2 被否决方案

- 前端内置管理密码：构建产物可直接提取，否决。
- 使用 visitor ID 作为管理员身份：visitor 可伪造，否决。
- 一期建设账号密码表：需要密码找回、会话、CSRF、成员管理，超出当前单管理员场景。
- 管理后台直接连接数据库：暴露数据库网络与凭据，否决。
- 先做纯图表不做反馈审核：无法形成运营闭环，否决。

### 2.3 风险控制

- Token 只保存于标签页会话，不写 localStorage、Cookie 或 URL。
- API 返回字段采用显式 DTO，不返回网络标识和内部哈希。
- 管理列表参数使用 Zod 校验，page size 最大 100。
- 审核状态只允许 pending/approved/rejected，备注最大 500 字。
- 管理 API 不计入公众 API 成功率，避免运维查询污染业务指标。

评审结论：产品、技术、安全、测试一致同意进入开发。
