# 27 岗位信息导入体验优化 PRD

## 1. 背景与问题

用户在招聘 App、网页和 JobLens 之间切换时，需要长按、全选、复制岗位 JD，再回到产品填写多个字段。移动端长文本选择困难，部分招聘平台也限制复制，导致用户在发起检测前流失。

当前首页将招聘平台、公司名称、岗位名称置于 JD 前方，且核心路径仅支持手动粘贴文本。目标是将首次检测从“整理并填写信息”缩短为“一次导入、一次确认”。

## 2. 目标与非目标

### 目标

- 提升岗位检测完成率与首次提交速度。
- 让用户优先导入岗位内容，补充字段不阻塞检测。
- 保留可编辑、可控的数据确认过程，不将错误内容直接提交分析。
- 不扩大招聘平台抓取和用户隐私风险。

### 非目标

- P0 不做招聘平台账号登录、自动抓取或通用岗位链接解析。
- P0 不上传图片、不接入 OCR。
- P0 不要求用户注册、安装扩展或安装 PWA。

## 3. 用户与场景

| 用户 | 场景 | 当前阻力 | 目标体验 |
|---|---|---|---|
| 移动端求职者 | 在招聘 App 查看岗位 | 长按与跨应用粘贴麻烦 | 复制后点一次“从剪贴板导入” |
| 桌面端求职者 | 在招聘网站浏览岗位 | 需要复制多段内容并补填字段 | 粘贴 JD 后自动识别可用字段 |
| 复制受限用户 | 岗位详情无法完整复制 | 无法获取原文 | 后续上传截图识别并确认 |

## 4. 方案与优先级

### P0：一键导入岗位文本

首页将首要区域改为“导入岗位信息”，包含以下能力：

1. 主按钮“从剪贴板导入”。用户点击后尝试读取纯文本剪贴板。
2. 成功后将文本写入 JD 输入框，显示字符数和成功反馈，并自动聚焦文本框。
3. 读取失败、无文本、权限拒绝或浏览器不支持时，不展示技术错误；改为聚焦文本框，并提示“请在招聘 App 或网页复制岗位内容后粘贴”。
4. JD 输入框仍支持系统原生粘贴、输入和修改。
5. 仅当 JD 非空时，显示“自动识别补充信息”。首版采用轻量启发式提取岗位名称、公司、平台；结果只预填，不覆盖用户已输入内容。
6. 将平台、公司、岗位名称和 HR 聊天记录放入“补充信息（选填）”可展开区域。默认只露出 JD 和提交按钮。

### P1：截图识别导入

在 P0 指标达标后，增加“上传岗位截图识别”：支持 1 至 3 张截图，OCR 后进入可编辑确认态，再提交检测。详细 OCR 研究与隐私约束见 `docs/08-screenshot-ocr-research.md`。

### P2：桌面浏览器扩展

用户在招聘网页主动点击扩展后，提取当前可见岗位文本并跳转至 JobLens 预填页。扩展不得后台采集网页、账号或聊天信息。

### 暂缓：岗位链接解析

不同招聘平台受登录、动态渲染、访问规则和反爬策略影响，稳定性与合规成本都较高。链接解析不进入当前版本。

## 5. 交互设计

```text
导入岗位信息
[ 从剪贴板导入 ]
或直接粘贴岗位 JD

[ JD 输入框：可编辑、展示字符数 ]
[ 已识别：岗位名称 / 公司 / 平台 ] 仅在有结果时展示

补充信息（选填，展开）
- 平台、公司、岗位名称
- HR 聊天记录

[ 开始检测 ]
```

### 状态

| 状态 | 触发 | 展示与操作 |
|---|---|---|
| 默认 | 初次进入 | 导入按钮、JD 输入框、折叠的补充信息 |
| 成功导入 | 剪贴板有文本 | 成功提示、JD 内容与字符数、自动聚焦 |
| 无内容 | 剪贴板为空 | 聚焦 JD，提示复制后重试 |
| 权限/兼容失败 | 浏览器拒绝读取 | 聚焦 JD，提示使用系统粘贴；不阻断输入 |
| 内容过长 | 超过前端限制 | 截断前提示，禁止提交直到用户删减 |
| 自动识别完成 | JD 已导入 | 可编辑的识别字段，不改变已有用户输入 |

## 6. 数据、隐私与合规

- 剪贴板只在用户明确点击后读取；不监听页面剪贴板变化，不自动读取。
- 仅读取纯文本，不读取图片、文件或富文本附件。
- 复用现有敏感信息检测、长度限制、匿名 visitor_id 和数据删除机制。
- 埋点只记录事件名、来源方式、字符长度分段和成功/失败原因分类；不得记录 JD 原文或剪贴板内容。
- P1 图片上传前必须提示敏感信息风险，并按既有隐私保留策略删除原图。

## 7. 埋点与验收指标

| 事件 | 属性 |
|---|---|
| `job_import_clicked` | `method=clipboard` |
| `job_import_result` | `method`, `result=success|empty|denied|unsupported`, `length_bucket` |
| `job_fields_autofilled` | `field_count` |
| `detect_submitted` | `entry_method=typed|pasted|clipboard`, `has_hr_chat` |

首轮灰度目标：

- “进入首页到提交检测”的中位时长下降 30%。
- 检测提交完成率相对基线提升 15%。
- 剪贴板导入成功后提交检测的转化率不低于手动粘贴路径。
- 因导入功能产生的前端错误率低于 1%。

## 8. 验收标准

- 支持 HTTPS 环境下 Chromium、Safari、Firefox 的正常与降级路径。
- 点击导入后，成功时 JD 写入输入框；失败时用户可立即手动粘贴，无死路。
- 导入不会覆盖用户已经填写的平台、公司、岗位名称或 HR 聊天记录。
- 所有中英文文案随现有语言选择器切换。
- 键盘、屏幕阅读器可触达导入按钮与折叠区域。
- 现有 JD 校验、敏感信息拦截、验证码与检测接口行为不回归。

## 9. 开发拆分

| 优先级 | 工作项 | 交付 |
|---|---|---|
| P0 | 前端导入组件 | 剪贴板读取、降级、状态与双语文案 |
| P0 | 首页信息架构调整 | JD 首屏、补充字段折叠、可编辑预填 |
| P0 | 轻量字段提取 | 本地启发式提取与不覆盖策略 |
| P0 | 埋点 | 导入与提交漏斗事件 |

## 10. 开发专家评审

### 结论

**P0 可实施，建议按“剪贴板导入 + 本地预填 + 折叠补充信息”上线。** 不需要调整现有 `POST /api/reports/detect` 协议，也不需要新增后端存储或第三方服务。

### 技术建议

- 使用 `navigator.clipboard.readText()`，只在按钮点击回调中调用；该 API 要求安全上下文且可能被权限策略或浏览器拒绝。
- 不要在首次进入页面时请求剪贴板权限，也不要实现自动轮询读取。
- 将剪贴板逻辑封装为独立组件或 hook，返回 `success`、`empty`、`denied`、`unsupported` 状态，便于测试。
- 以 `try/catch` 统一处理 API 不存在、权限拒绝和系统失败；降级为原生 textarea 粘贴。
- 字段自动提取先使用前端规则，不为该功能额外调用大模型，避免成本、延迟与把剪贴板内容提前发送到第三方。
- P0 只读取文本；截图 OCR 需要新后端接口、文件校验、限流、异步任务、原图删除与测试，独立进入 P1。

### 风险与处置

| 风险 | 处置 |
|---|---|
| Safari 或内嵌浏览器拒绝剪贴板 | 明确降级到原生粘贴，不能把读取失败视为产品错误 |
| 剪贴板内容过长或非 JD | 导入后显示长度，保留现有提交校验与用户编辑权 |
| 自动提取误判 | 只预填空字段，标记为“已识别，可修改” |
| 中文招聘 App 文本格式不统一 | 规则仅做弱辅助，不将识别结果作为必填或风险结论 |
| 隐私与埋点泄露原文 | 仅记录长度分段和结果分类，不上传原文 |

### 测试清单

- 有文本、空文本、拒绝权限、不支持 API、异常抛出五种导入结果。
- 导入文本后手动修改、清空、重新导入。
- 已填写字段不被自动预填覆盖。
- 中文和英文页面文案、键盘焦点、移动端布局。
- 现有手动粘贴、提交、敏感信息拦截和验证码回归。

### 进入开发的门槛

1. 确认 P0 不包含截图上传与岗位链接解析。
2. 确认事件名称及隐私字段，接入现有日志策略。
3. 设计稿明确按钮、错误提示和补充信息展开态。
4. 通过前端单元测试与生产构建后再灰度发布。
